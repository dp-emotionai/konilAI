## 2. Backend: вынести и довести до «продакшена»

### 2.1 Что предлагается

- **Структура**: по желанию вынести `backend/` в отдельную папку `elas-backend` рядом с `elas-frontend` (или оставить внутри фронта — тогда один репозиторий). Рекомендация: вынести для чёткого разделения и деплоя.
- **PostgreSQL + Prisma**:
  - Добавить PostgreSQL как основную БД.
  - Ввести Prisma: `schema.prisma`, миграции, `PrismaClient`.
  - Модели минимум:
    - **User**: id, email, role (enum), passwordHash, createdAt (и при необходимости name, status).
    - **Group**: id, name, teacherId (FK), createdAt.
    - **Session**: id, title, type (lecture | exam), status (draft | active | finished), groupId, createdById (teacher), startedAt, endedAt, createdAt, code (уникальный код для входа).
    - **ConsentRecord** (по ТЗ — явное согласие): id, userId, sessionId, consentedAt, ipHash или без PII — только факт согласия для сессии.
- **Auth**:
  - **POST /auth/register**: email, password, role (или только student по умолчанию); хэш пароля (bcrypt), сохранение User в БД.
  - **POST /auth/login**: email + password → проверка bcrypt → выдача JWT (access + опционально refresh) или session cookie.
  - Не хранить пароли в открытом виде; использовать только bcrypt (или argon2).
- **REST для групп и сессий** (под текущий фронт):
  - **Groups**: GET /groups, GET /groups/:id, POST /groups (create), PATCH /groups/:id. Привязка к teacher (createdById).
  - **Sessions**: GET /sessions, GET /sessions/:id, POST /sessions (create), PATCH /sessions/:id (start/end, смена status). Связь с Group через groupId.
  - Ответы в формате, совместимом с текущими типами на фронте (Session, GroupSession, Group).
- **Сигналинг**: оставить текущий WebSocket `/ws`; при необходимости проверять JWT при `join` (sessionId + role) и привязывать к сессии в БД (проверка, что сессия active и пользователь имеет право).

### 2.2 Чем лучше

- Один источник правды: группы и сессии живут в БД, а не в моках.
- Реальная авторизация и роли — основа для диплома и соответствия ТЗ (роли Студент/Преподаватель/Администратор).
- Consent в БД — явный аудит согласия (этика и GDPR).
- Готовность к интеграции с Python ML: сессии с id и status; позже — вызовы start/stop и привязка аналитики по sessionId.

### 2.3 Как лучше сделать

- В backend: `npm i prisma @prisma/client bcrypt jsonwebtoken`, `npm i -D @types/bcrypt @types/jsonwebtoken`.
- Prisma: `prisma init`, описать схему, `prisma migrate dev`.
- Роуты разнести по файлам: `auth.ts`, `groups.ts`, `sessions.ts`; middleware для проверки JWT и роли.
- Конфиг: `DATABASE_URL`, `JWT_SECRET`, `CORS_ORIGIN` из env.

## 3. Frontend: подключение к реальному backend

### 3.1 Что предлагается

- **Env**: `NEXT_PUBLIC_API_URL=http://localhost:4000` (или другой base URL).
- **Замена вызовов**:
  - Вместо импортов из `@/lib/mock/*` в `src/lib/api/teacher.ts` и `src/lib/api/student.ts` — вызовы `fetch(NEXT_PUBLIC_API_URL + '/sessions', { headers: { Authorization: 'Bearer ' + token } })` и т.п.
  - Сохранить те же имена функций и типы возврата (или минимальные адаптеры), чтобы страницы и компоненты не переписывать.
- **Auth**:
  - Страницы логина и регистрации: формы, валидация (email, пароль), вызов POST /auth/login и POST /auth/register.
  - После успеха: сохранять токен (например в httpOnly cookie через backend или в localStorage для простоты демо), сохранять роль из ответа в state/context и редирект на ROLE_HOME[role].
  - Кнопки «Demo Teacher / Student / Admin» можно оставить для локальной демо-режима (если API недоступен) или убрать после стабилизации API.
- **Группы и сессии**: все списки и детали — с backend; индикаторы загрузки и ошибок оставить/добавить там, где сейчас скелетоны.

### 3.2 Чем лучше

- Единый источник данных: нет расхождений между «демо» и реальностью.
- Реальный сценарий для диплома: регистрация → вход → создание группы/сессии → старт сессии → студент подключается по коду.
- Минимальные правки UI: только смена источника данных и добавление обработки ошибок/тостов.

### 3.3 Как лучше сделать

- Вынести базовый URL в `src/lib/api/client.ts`: getAuthHeaders(), request GET/POST/PATCH.
- В teacher/student API — вызывать этот client; типы оставить в `src/lib/types.ts` или продублировать DTO с backend при необходимости.
- Роутинг: если нет токена и страница защищена — редирект на /auth/login.

## 4. Information Architecture (IA) по ролям

### 4.1 Что предлагается

- **Teacher**: меню/навигация — Dashboard, Groups, Sessions, Reports, Compare; хлебные крошки на подстраницах; кнопки «Back to Groups» / «Back to Session» где уместно.
- **Student**: Dashboard, Sessions (мои сессии), Summary (личная сводка), Privacy (инфо + согласие); хлебные крошки.
- **Admin**: Dashboard, Users, Groups, Model, Storage, Audit; хлебные крошки.
- Общий компонент Breadcrumbs и использование на всех ключевых страницах.

### 4.2 Чем лучше

- Соответствие ТЗ (чёткое разделение по ролям) и лучшая навигация для диплома/презентации.
- «Back to…» уменьшает количество кликов и путаницу.

### 4.3 Как лучше сделать

- В `TopNav` или в `Shell` — пункты меню по роли (уже частично есть); дополнить недостающие ссылки.
- Один компонент `Breadcrumbs` (например в `components/layout`), принимающий массив `{ label, href }`; на страницах передавать контекст (Group name, Session title и т.д.).

## 5. Design tokens

### 5.1 Что предлагается

- Вынести в одно место (например `globals.css` или отдельный `tokens.css`):
  - Цвета: primary (purple), background, surface, border, text primary/secondary, success/warning/error.
  - Типографика: размеры шрифтов, веса, line-height для заголовков и тела.
  - Радиусы, тени, отступы для Card, GlassCard, Table, Badge, Button.
- Компоненты Card, GlassCard, Table, Badge, Button — использовать эти токены (CSS-переменные или Tailwind-классы из одного источника).

### 5.2 Чем лучше

- Единый визуальный язык; проще менять тему (light/dark уже есть — токены упростят поддержку).
- Меньше дублирования и «магических» значений в компонентах.

### 5.3 Как лучше сделать

- В Tailwind v4: в `globals.css` задать переменные (например `--color-primary`, `--font-heading`), в `tailwind.config` или в `@theme` подставить их. Компоненты перевести на классы из токенов.
- Патч-style: менять по одному компоненту, не ломая текущий вид.

## 9. Интеграция с Python ML (позже)

### 9.1 Что предлагается (без реализации в первом этапе)

- Python-сервис (FastAPI): эндпоинты для старта/остановки анализа сессии, приём кадров (или метаданных), возврат агрегированной аналитики и отчётов.
- Node backend: прокси или отдельные вызовы к Python:
  - POST /sessions/:id/start → вызов Python start; PATCH /sessions/:id (end) → вызов Python stop.
  - GET /sessions/:id/summary → прокси к Python или чтение из БД (если Python пишет в общую БД/Redis).
  - WebSocket для realtime: либо прокси WS к Python, либо Python шлёт в Node, Node ретранслирует клиентам.
- Frontend: teacher live monitor и analytics/reports — получать данные из этих API/WS, без переписывания страниц.

### 9.2 Чем лучше

- ТЗ рекомендует Python (FastAPI) для ML и Computer Vision; Node остаётся оркестратором, auth и REST для фронта.
- Чёткое разделение: распознавание эмоций и метрики — в Python; пользователи, сессии, согласие — в Node + PostgreSQL.

## 10. Порядок работ (рекомендуемый)

1. **Backend: PostgreSQL + Prisma + Auth + REST для groups/sessions**  
   Сначала поднять БД и API, чтобы фронт мог к ним подключиться.

2. **Frontend: env, API client, замена моков на HTTP, логин/регистрация**  
   Подключить реальный логин и данные; оставить демо-кнопки опционально.

3. **IA: меню по ролям, хлебные крошки, «Back to…»**  
   Небольшие правки навигации и один компонент Breadcrumbs.

4. **Design tokens**  
   Постепенно перевести глобальные стили и компоненты на токены.

5. **Product Tour**  
   Проверить/дополнить блок «3 steps» на Teacher Dashboard.

6. **Privacy / Ethics**  
   Якоря и при необходимости контент.

7. **Бейджи Mock**  
   Пока данные с API — помечать обратное (или убрать Mock там, где уже Live).

8. **Интеграция Python ML**  
   После стабилизации backend и фронта — отдельным этапом.

## 11. Итог

- **Сначала** имеет смысл сделать пункты **1 и 2**: реальный backend (PostgreSQL, Prisma, auth, REST) и подключение фронта к нему. Это даёт полноценный сценарий «логин → группы → сессии → старт/подключение» и основу для диплома.
- **Далее** — пункты 3–7 (IA, токены, тур, Privacy/Ethics, бейджи) без изменения большой логики страниц.
- **Потом** — интеграция с Python ML по описанной схеме.

Если нужно, могу следующим шагом расписать конкретные изменения по файлам (например схема Prisma, список эндпоинтов и поля ответов, или правки в `teacher.ts`/`student.ts` под fetch).
